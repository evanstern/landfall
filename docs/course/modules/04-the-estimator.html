The Estimator That Won’t Be Fooled

The one learning part of landfall — and the strict rules that keep it honest

Where does “time-to-land” come from?

The Gate multiplies velocity by predicted call time — but who predicts the call time? Think of how you predict your commute: the trip has a distance (fixed) and a pace (changes with traffic). landfall splits latency the same way:

📏

Points — the distance. How big the ask is. A property of the question, fixed at registration.

×
🚦

Seconds-per-point — the pace. How slow this oracle currently is. Measured live by the Estimator.

=
⏱️

Time-to-land — the prediction the Gate multiplies by velocity.

💡
Aha! Factoring means new decisions start smart

Because size and speed are separate, a brand-new decision class needs zero timing history — assign it points and it inherits the oracle’s measured pace, the way you can estimate a brand-new errand across town because you already know the traffic. One estimator per oracle, not one per decision.

A memory that bends but doesn’t break

You know your commute is about 30 minutes. One day a crash makes it 90 — you don’t start planning 90 every day. But when new roadwork adds 5 minutes every day, your gut slowly updates. That exact instinct is six lines of estimate.go:

estimate.go 

	spike := secPerPoint > e.cfg.SpikeFactor*e.estimate
	if spike {
		e.spikes++
	} else {
		e.estimate = (1-e.cfg.Alpha)*e.estimate + e.cfg.Alpha*secPerPoint
	}
PLAIN ENGLISH

Is this new timing more than 3× what we believe? Then it’s a spike — the freak traffic jam.

If it’s a spike…

…count it (we never forget it happened) but do NOT let it move the estimate.

Otherwise, it’s a normal day…

…so blend it in gently: keep 80% of the old belief, adopt 20% of the new reading.

Freak events can’t poison the estimate; steady change still steers it.

That 80/20 blend is called an EWMA. Old evidence never vanishes; it just fades.

The rules of evidence

Two strict doctrines keep this witness honest — both are one-sentence rules you can demand in any system you build:

✅

Learn from successes only

A failed call’s duration says nothing about how long a good answer takes. Failures go to the circuit breaker, never the timing model.

🐢

When ignorant, assume slow

A fresh estimator is seeded pessimistically — assumed slower than reality. An uncalibrated system refuses calls and falls back, rather than acting on stale answers.

And the tuned numbers ship right in the file — carried over from the project this library was distilled from:

estimate.go 

func DefaultEstimatorConfig() EstimatorConfig {
	return EstimatorConfig{Alpha: 0.2, SpikeFactor: 3.0, WindowSize: 20, BreachRate: 0.3}
}
PLAIN ENGLISH

The battle-tested default settings, as one ready-made bundle:

Blend 20% of each new reading; call anything over 3× the estimate a spike; watch the last 20 samples; and worry when more than 30% of them are spikes.

Done — hosts start here and only change these numbers deliberately.

The smoke alarm

One freak jam is noise. But if 6 of your last 20 commutes were “freak” jams… they’re not freak anymore — your whole mental map is stale. The Estimator watches its recent window, and when the spike rate crosses 30% it raises a one-time signal: recalibrate me.

1
Spikes are excluded, but counted

Every sample marks a slot in a rolling 20-slot window: spike or normal.

2
The window rate is the tell

Spike rate above 30% over a full window means the baseline itself is lying — the signal fires once, and re-arms only after things recover.

3
A human moves the baseline

The recorded calibration is only ever changed by a person. The live estimator adapts; the official record never drifts silently.

🧑‍⚖️
Machines adapt, humans commit

This split — a live estimate that adapts freely, a recorded baseline only a human may change — shows up in great production systems everywhere. It’s a pattern worth asking your AI tools for by name: “let it adapt at runtime, but never let it rewrite its own configured baseline.”
