The Flight

Takeoff clearance is not landing clearance — how an allowed call is checked again when it returns

Cleared for takeoff ≠ cleared to land

An airplane gets two separate clearances: one to take off, and a fresh one to land — because the weather at the destination may have changed during the flight. landfall works exactly the same way. The Gate’s “allow” was a prediction; while the oracle thinks, the world keeps moving.

1
Checkout — takeoff

The moment an allowed call launches, a Lease freezes a snapshot: which class flew, the budget it was allowed against, the drift the Gate bet on, and the world’s generation number.

2
Flight — the world moves on

The oracle thinks. Meanwhile drift accumulates, and if something truly major happens, the host bumps the generation counter.

3
Land — the second clearance

When the answer arrives, Land compares the frozen snapshot against reality and rules: apply it, or don’t — with the numbers to prove it.

The flight recorder: Checkout

The entire takeoff procedure is three lines in lease.go:

lease.go 

func Checkout(v Verdict, gen uint64) Lease {
	return Lease{Gen: gen, Class: v.Class, Budget: v.EffectiveBudget, PredictedDrift: v.PredictedDrift}
}
PLAIN ENGLISH

At launch, take the allowed Verdict and the current generation number…

…and freeze four facts into the Lease: the generation at takeoff, who flew, the budget it was judged against, and the drift the Gate predicted.

That’s it — a tiny, tamper-proof boarding pass.

🧊
Why freeze the prediction inside the lease?

Because carrying the bet with the flight means every landing can compare “what the Gate predicted” against “what actually happened” in one record — no digging through old logs to match them up. Databases call this style optimistic concurrency control: proceed hopefully, validate at the end.

Three ways to land

✅

Landed

Same generation, drift within budget. The world still matches — apply the answer.

🔄

Superseded

A salient event bumped the generation mid-flight. Don’t apply it — but you may quietly warm a cache with it.

⏰

Stale

No supersession, but actual drift blew past the budget — the Gate’s prediction was wrong. Record it loudly.

And the landing check itself, verbatim from lease.go:

lease.go 

func (l Lease) Land(currentGen uint64, actualDrift float64) Landing {
	ld := Landing{
		Class: l.Class, Gen: l.Gen, CurrentGen: currentGen,
		Budget: l.Budget, PredictedDrift: l.PredictedDrift, ActualDrift: actualDrift,
	}
	switch {
	case currentGen != l.Gen:
		ld.Outcome = Superseded
	case actualDrift > l.Budget:
		ld.Outcome = Stale
	default:
		ld.Outcome = Landed
	}
	return ld
}
PLAIN ENGLISH

The landing clearance: hand in the world as it is now — current generation and the drift that actually accumulated.

Start writing the Landing record…

…copying in everything frozen at takeoff…

…plus the prediction and what really happened — side by side in one record.

Snapshot complete.

Now the ruling — checked in strict priority order:

Did the generation change during the flight?

Then something big happened: Superseded — no matter how small the drift was.

Otherwise, did real drift exceed the budget?

Then the Gate’s bet lost: Stale.

Neither?

Safe to apply: Landed.

End of the ruling.

Return the record — every landing gets one, even the failures. Especially the failures.

Done: pure arithmetic, same inputs, same outcome, every time.

💡
Aha! Supersession trumps staleness

Notice the switch order: the generation check comes first. If the big event happened, the answer is dead regardless of how little drift accrued — and every dropped answer leaves a record. A silently dropped answer is how “why is the assistant so passive?” mystery bugs are born.

The report card that writes itself

Because every Landing carries PredictedDrift and ActualDrift in the same units, the landing log doubles as the Gate’s report card — for free.

predicted 4.5 / actual 4.1 Healthy: the Gate’s bets are landing close to reality
predicted 4.5 / actual 9.8 One bad bet — noise happens, keep watching
predicted < actual, every day A pattern of Stale outcomes: the velocity measurement or the timing calibration is lying — a human should fix the inputs
